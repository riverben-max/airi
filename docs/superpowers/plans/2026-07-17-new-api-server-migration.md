# New API 服务器迁移实施计划

## 背景与迁移边界

AIRI 当前使用旧服务器上的 `new-api-2` 容器，通过宿主机 3001 端口及
`api.aifamily.vip`、`aifamily.qplus.pub` 反向代理提供服务。该实例使用 MySQL
数据库 `newapi2`，不以容器内 SQLite 文件作为主数据源。

本次将该实例完整复制到 `8.137.153.77`，并让 `http://8.137.153.77/` 可直接
访问。旧服务器保持在线，禁止停止容器、修改 Nginx、修改 DNS、修改数据库权限或
覆盖任何源数据。旧机上的另一套 SQLite `new-api` 实例不属于 AIRI 当前链路，
不在本次迁移范围内。

## 实施范围

1. 记录源容器的镜像摘要、环境变量、重启策略和数据库结构，使用
   `mysqldump --single-transaction` 创建在线一致性快照。
2. 备份目标机现有 Nginx 配置和 MySQL 状态；默认不修改或重建 `kiro-rs` 容器。
3. 在目标机现有 MySQL 中创建独立的 `newapi2` 数据库和最小权限用户，导入完整
   数据库快照并核对表数、关键表行数和数据大小。
4. 直接拉取源实例相同摘要的预构建 New API 镜像并启动目标容器。服务器上禁止
   执行源码构建、`docker build`、pnpm、Node 构建或其他编译任务。容器采用 host network，
   仅由目标机 Nginx 通过 `127.0.0.1:3000` 访问，MySQL 继续仅监听本机。
5. 将目标机根路径代理到 New API，同时保留 `/kiro-rs/` 管理入口与现有长连接、
   模型接口代理行为。
6. 验证 New API 首页、状态 API、登录及带鉴权的模型接口。资源足够时确认
   `/kiro-rs/admin` 仍可访问；若部署期间确认内存不足，可在备份配置后停止并删除
   `kiro-rs` 及其根路径代理。

## 数据一致性

- 源服务不停机，因此迁移得到的是 `mysqldump` 开始时的一致性快照；导出完成后
  旧实例新增的数据不会自动双写到目标机。
- 导出文件仅临时保存在迁移目录，传输和导入完成后保留一份受限权限备份用于回滚，
  不写入仓库。
- 数据库口令只存放在目标机权限为 `0600` 的环境文件中，不出现在提交、日志或
  交付说明中。

## 验收标准

- 目标 New API 容器使用与源实例一致的镜像摘要，状态为 running，重启策略生效。
- 目标 MySQL 仍只监听 `127.0.0.1`，New API 数据库表数和关键表行数与快照一致。
- `http://8.137.153.77/` 返回 New API 页面，核心 API 能读取迁移后的用户、令牌、
  渠道和系统选项。
- 默认保留 `http://8.137.153.77/kiro-rs/admin`；若因内存不足按授权删除，则记录
  删除前备份及删除后的可用内存。
- 旧服务器的 New API、Nginx、MySQL 和 DNS 均未修改，旧入口继续可用。

## 回滚

若目标 New API 或 Nginx 验收失败，恢复目标机迁移前的 Nginx 配置并 reload，停止
并删除目标 New API 容器。目标数据库保留用于排查或从迁移前目标 MySQL 备份恢复；
回滚不涉及旧服务器，因为旧链路始终保持在线。

## 实施结果

2026-07-17 已完成迁移。源库通过在线一致性快照导出，目标导入 27 张表；用户、
令牌、渠道和系统选项的精确行数与快照一致。目标容器使用源实例的预构建镜像摘要
`sha256:39ddc868b8ee81218dd6159c9ba8c1225e77980869b8e473634e78d422a0353a`，
服务器上未执行任何构建。

目标 New API 通过 `127.0.0.1:3000` 接受 Nginx 流量，并通过 Unix socket 访问仅
监听本机的 MySQL。容器重启后根页面、状态 API 和带有效令牌的模型接口均返回
200，模型列表包含 2 项；`/kiro-rs/admin` 继续返回 200。整机 swap 未使用，
`kiro-rs` 无需删除。旧服务器容器和 API 保持在线，旧 Nginx 配置未修改。
