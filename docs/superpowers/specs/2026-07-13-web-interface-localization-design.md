# Web 用户界面中文化设计

## 目标

让简体中文环境下用户实际可见的 Web 界面大部分恢复为中文，优先覆盖设置、聊天、导航、弹窗、提示、空状态、认证和管理界面，同时避免翻译产品名、协议名和必须原样输入的技术标识。

## 现状与根因

- `packages/i18n/src/locales/zh-Hans/settings.yaml` 相比英文语言包缺少 215 个标量键。Vue I18n 的回退语言为英文，因此这些位置会直接显示英文。
- 简体中文 `settings.yaml` 中还有约 324 个含英文字母且与英文值完全相同的条目，其中既有应翻译的普通界面文案，也有应保留的品牌名和技术术语。
- 设置及共享界面组件仍存在直接写在 Vue 模板或脚本中的英文文案，无法通过切换语言变成中文。
- 之前的修复集中处理少量已发现页面，没有建立语言包键完整性检查；后续功能继续加入后，漏译再次累积。

## 已选择方案

采用结构化本地化修复：

1. 以英文语言包的键结构为基准，补齐简体中文中用户可见的缺失键。
2. 逐项审查简体中文语言包中仍为英文的值，翻译普通界面文案，并保留确有必要的专有名词和技术标识。
3. 将生产界面中写死的英文迁移到现有 i18n 结构，为英文和简体中文同时提供对应键。
4. 增加语言包键完整性回归检查，防止新增英文键在简体中文中静默回退。

不采用直接把 Vue 英文替换为中文的方式，以免英文及其他语言界面也被强制显示中文；也不只补语言包，因为写死在组件中的英文仍会继续漏出。

## 范围与优先级

按用户可见程度依次处理：

1. `packages/stage-pages` 中当前 Web 设置页及其弹窗、向导、表单、提示和空状态。
2. `packages/stage-layouts` 和 `packages/stage-ui` 中聊天、导航、角色操作及设置页实际复用的组件。
3. `apps/stage-web` 和 `apps/ui-server-auth` 中用户能够进入的主页、认证、账户和管理界面。
4. 简体中文语言包里上述界面依赖的条目。

若某段英文只存在于开发工具、Storybook、测试夹具、日志、内部提示词、不可达旧页面或构建产物中，本轮不处理。若修改复杂内部流程才能替换一段用户通常不可见的英文，也不扩大本轮范围。

## 术语边界

保留 OpenAI、WebSocket、MCP、API、URL、JSON、Flux、Live2D、VRM、模型名称、Provider 名称和精确配置键等原文。普通说明仍翻译为中文，例如保留术语但翻译其上下文。用户必须照抄或与外部服务对应的值不做本地化。

## 实施约束

- 沿用现有 `vue-i18n`、`useI18n()` 和 YAML 命名层级，不引入新的翻译框架。
- 优先复用已有键；只有语义不同或当前没有合适键时才新增。
- 不修改业务逻辑、路由行为、Provider 配置结构或组件视觉样式。
- 所有中文文件保持 UTF-8 无 BOM 和 LF。
- 不修改 `apps/stage-web/dist` 等生成文件，生产构建负责重新生成产物。

## 验证策略

- 增加简体中文与英文核心语言包的键完整性测试；允许明确排除非 Web 或尚未支持的内部命名空间，但排除项必须具体。
- 对迁移到 i18n 的关键组件做源码契约或渲染测试，确认中文键存在且组件不再直接输出对应英文。
- 解析所有修改过的 YAML，确认语法正确、插值变量和英文版本一致。
- 运行 `packages/i18n`、相关前端包的定向测试与类型检查，并使用项目 Node 24 执行 Stage Web 生产构建。
- 构建后在简体中文环境抽查设置主页、主要设置子页、聊天控件、弹窗、认证和管理入口；记录仍保留英文的专有名词或非目标区域。

## 验收标准

- 简体中文环境下主要用户路径不再因缺键回退为整句英文。
- 设置、聊天、导航、弹窗、提示和空状态中的普通英文文案大部分已中文化。
- 保留的英文可解释为品牌、模型、协议、技术字段或明确非目标区域。
- 英文及其他语言仍通过 i18n 正常显示，不被硬编码中文污染。
- 语言包完整性检查、相关测试、类型检查和 Stage Web 生产构建通过。

## 非目标

- 不翻译开发工具、Storybook、测试数据、日志、LLM 提示词和第三方内容。
- 不翻译品牌名、模型名、Provider 名称、协议缩写和精确 API 字段。
- 不为了极少出现的内部英文重构复杂业务代码。
- 不处理视觉改版、交互重构或与本地化无关的问题。
- 不进行生产部署，除非用户另行要求。
